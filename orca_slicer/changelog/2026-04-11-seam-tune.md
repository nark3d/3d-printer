```
INHERITED (not explicit in profile — falls back to code defaults):
  scarf_joint_speed: 100 mm/s            ← too fast, community says <100
  seam_slope_start_height: 0             ← too low, Bambu uses 10%, Teaching Tech 50%
  seam_slope_min_length: 20 mm           ← matches default (lock in)
  scarf_angle_threshold: 155°            ← matches default (lock in)
  scarf_overhang_threshold: 40%          ← matches default (lock in)
  seam_slope_entire_loop: false          ← matches default (lock in)
  wipe_on_loops: false                   ← matches default but should be explicit
  wipe_before_external_loop: false       ← matches default but should be explicit
```

**Three high-impact problems**:

1. **`seam_gap: 5%`** — contradicts the strongest community consensus. Noisyfox (PR #3839 author): "seam gap should not matter when this is enabled." begna112 (issue #7326, Sunlu PLA+ X1C): "Disabling scarf seam gaps seems to mostly negate the inner scarf issue and lead to better quality." FuryriderX (PR #3839 PETG tester): "I had to set seam gap to 0." Nobody who uses scarf has 5%.

2. **`scarf_joint_speed: 100 mm/s` (inherited)** — at our outer_wall_speed of 100 mm/s, the default 100% multiplier means scarf runs at exactly the upper limit of Adam L's 75-100 mm/s range. No safety margin. Community consensus says the sweet spot is the middle of this range.

3. **`seam_slope_start_height: 0` (inherited code default)** — Bambu uses 10% as their default, Teaching Tech uses 50%. At 0, the first scarf segment starts at ~0.02 mm above the previous layer — almost touching it. Any tiny filament droplet on the nozzle tip gets squashed into the previous layer, creating a visible blob.

## Changes in this pass (9 total)

### High-impact change (fixes the regression most likely)

### 46. `seam_gap`: `"5%"` → `"0%"`

**Research sources**:
- Primary: [`orca_slicer/process/seams_deep_dive.md`](../process/seams_deep_dive.md) § 3.2 (source code analysis), § 11.2 (begna112), § 12.7 (Noisyfox + FuryriderX), § 21
- Noisyfox PR #3839 comment: "seam gap should not matter when this is enabled, as I've already disabled the seam gap for external perimeter when scarf is used. If seam gap does make a difference in your test, then that's a bug."
- begna112 issue #7326 (Sunlu PLA+ X1C HF, 0.4 mm nozzle, line width 0.42 mm): "Disabling scarf seam gaps seems to mostly negate the inner scarf issue and lead to better quality." Before/after photos showed near-imperceptible seam with `seam_gap: 0`, visible bump with `seam_gap: 10%`.
- FuryriderX PR #3839 first-tester (PETG): "I had to set seam gap to 0 and disable wipe on loops and wipe on external loops." Top sample with those settings: near-invisible seam.
- vgdh discussion #4325: "start height 0.1 and seam gap = 0" — recommended for first-layer adhesion issues on scarf

**Why**: the single most consistently-reported fix for "visible scarf seam" when scarf is otherwise correctly configured. When scarf is enabled, the OrcaSlicer code applies `seam_gap` at the END of the scarf slope via `clip_slope(seam_gap)` — clipping a fraction off the scarf ramp's tail. For a non-zero seam_gap, this creates a micro-step at the scarf end that shows as a visible artifact.

**Why 0% specifically**: per Noisyfox's code analysis, seam_gap should be literally irrelevant for scarf-external perimeters because the code sets `clip_length = 0` before the slope's own gap application. Setting it to 0 in the profile makes the intention explicit and prevents the clip_slope() secondary application from affecting the external perimeter scarf tail.

**Impact**: the scarf slope's tail will land cleanly without the micro-step clip. This is the fix most directly attested to by real-user empirical testing.

**Risk**: the traditional-seam fallback (for perimeters shorter than 20 mm or with corners sharper than 155°) will have `seam_gap: 0%` too. On sharp corners where traditional seams fire, this may create slightly less pressure buffering. But corners naturally hide seams, so the loss is minimal. Trade-off: scarf quality on 95% of the print improves; traditional fallback on the 5% of corners is marginally worse.

**Trade-off acknowledged**: for a traditional-seam-only profile, the community consensus is 5-15% (VzBot bundled 2%, Creality Sermoon 6%, OrcaSlicer default 10%, mjonuschat Voron 15%). Since we're scarf-dominant, 0% is the right call. See the seams deep-dive § 22 for the full analysis.

### Medium-impact changes (quality improvements)

### 47. Add `seam_slope_start_height`: (inherits 0) → `"10%"`

**Research sources**:
- Primary: [`orca_slicer/process/seams_deep_dive.md`](../process/seams_deep_dive.md) § 5.6, § 17.1
- Bambu `fdm_process_common.json` ships with `seam_slope_start_height: 10%` as their baseline
- Teaching Tech uses 50% (0.1 mm on 0.2 mm layers) per his YouTube video collaboration with Adam L
- vgdh discussion #4325: "start height 0.1 and seam gap = 0"
- Our own field notes Case 4: "Changed `seam_slope_start_height` to 50%. Reduced to a small bump at scarf entry."

**Why**: at the code default of 0, the first scarf segment starts at ~0.02 mm above the previous layer (for 10 steps on 0.2 mm layer height). That's almost zero clearance between the nozzle and the previous layer. Any residual filament droplet on the nozzle tip from retraction gets squashed into the previous layer, creating a visible blob at the scarf entry.

At `start_height: 10%`, the first segment starts at 0.02 + 10% × 0.2 = 0.04 mm — double the clearance. The droplet has more room to form cleanly before the nozzle contacts the previous layer.

**Why 10% specifically (vs Bambu's 10% or Teaching Tech's 50%)**: 10% is the Bambu default — a conservative starting value validated across millions of prints. Teaching Tech's 50% is more aggressive and was used to fix a specific "knotty blob" issue. 10% is the safer baseline; we can move to 50% if we still see entry blobs after this change.

**Impact**: less likely to produce an entry blob at the scarf ramp start. Expected visible improvement at scarf entry points.

**Risk**: minor. Values between 10% and 50% are all community-validated; going too high (>50%) creates a visible step instead of a ramp.

### 48. Add `scarf_joint_speed`: (inherits 100 mm/s) → `"75%"`

**Research sources**:
- Primary: [`orca_slicer/process/seams_deep_dive.md`](../process/seams_deep_dive.md) § 5.5
- Adam L's guide: "Optimal speeds are generally in the range 50-100 mm/sec for the scarf speed. Critical for avoiding wall distortions."
- Adam L's testing: 75 mm/s OK, 100 mm/s better, 150 mm/s "far worse"
- OrcaSlicer wiki: "Recommended to print scarf joints at a slow speed (less than 100 mm/s)"
- Anycubic Kobra S1 bundled profile ships `scarf_joint_speed: 35` (absolute mm/s)
- Our own 0.12mm Fine profile uses `scarf_joint_speed: 50%` explicitly
- Noisyfox PR #3839 photos explicitly show that lower outer wall speed dramatically improves scarf quality

**Why**: at our current `outer_wall_speed: 100` mm/s with the inherited default `scarf_joint_speed: 100%` multiplier, scarf runs at exactly 100 mm/s — the upper limit of Adam L's sweet spot. No safety margin. The community sweet spot is the middle of the 50-100 range.

**Why 75% specifically**: 75% × 100 mm/s = 75 mm/s, which is exactly in the middle of Adam L's "optimal range". Slower would be safer for quality but not necessary. Going to 50% (like our Fine profile) would be more conservative but probably unnecessary at 0.2 mm layer height.

**Impact**: scarf ramp runs at 75 mm/s instead of 100 mm/s. Gives the hotend more time to stabilise flow at the start and end of the ramp, reducing flow transients that cause visible artefacts.

**Risk**: minor print time increase (scarf region takes ~25% longer, but scarf is a tiny fraction of total print time). No quality risk.

### Low-impact changes (lock in inherited defaults as explicit, make profile self-documenting)

### 49. Add `seam_slope_min_length`: (inherits 20) → `"20"` explicit

**Research sources**: OrcaSlicer `PrintConfig.cpp` code default is 20; our profile inherits this implicitly.

**Why**: Adam L tested 20 mm and found "making the scarf length shorter than the default 20 starts to become a particular quality problem. I recommend leaving it at default of 20." Making this explicit locks in the value against future parent profile changes. Bambu uses 10 mm; Anycubic/Phrozen use 10 mm; Creality Sermoon uses 12 mm — lots of profiles shorten this, but Adam L's testing recommends 20 for quality on large features.

**Impact**: none (value unchanged). Just makes the profile self-documenting.

**Risk**: none.

### 50. Add `scarf_angle_threshold`: (inherits 155) → `"155"` explicit

**Research sources**: OrcaSlicer code default 155°; Adam L recommends 155° for production.

**Why**: locks in the value. Perimeters with corners more obtuse than 155° get scarf; sharper corners use traditional seams. Adam L's production recommendation.

**Impact**: none (value unchanged).

**Risk**: none.

### 51. Add `scarf_overhang_threshold`: (inherits 40%) → `"40%"` explicit

**Research sources**: OrcaSlicer code default 40%; community and Adam L recommend 40% as the default.

**Why**: locks in the value. Scarf is skipped where overhang exceeds 40%.

**Impact**: none.

**Risk**: none. **Caveat**: Issue #12270 reports this setting isn't always respected (scarf fires on overhangs regardless). Making it explicit doesn't fix the bug but documents the intent.

### 52. Add `seam_slope_entire_loop`: (inherits false) → `"0"` explicit

**Research sources**: OrcaSlicer code default false; Adam L "has not significantly improved results"; Issue #5465 reports causes volumetric flow drops.

**Why**: locks in the value. Enabling this would dramatically increase print time and has been empirically shown to not improve quality.

**Impact**: none.

**Risk**: none.

### 53. Add `wipe_on_loops`: (inherits false) → `"0"` explicit

**Research sources**:
- Adam L's "avoid" list: "Using Wipe on loops makes things bad" (with scarf)
- FuryriderX PR #3839: confirmed in first scarf testing
- shyblower issue #7326: confirmed on 2.2.0 — wipe_on_loops enabled on 2.1.1 worked fine, on 2.2.0 made things worse
- The OrcaSlicer wiki RECOMMENDS wipe_on_loops for general use, but Adam L's scarf-specific testing says OFF

**Why**: locks in Adam L's "must-avoid" recommendation. Our profile inherits `wipe_on_loops: false` from the parent chain (matches code default), but making it explicit is a zero-risk cleanup that survives future parent profile changes.

**Impact**: none (value unchanged).

**Risk**: none.

### 54. Add `wipe_before_external_loop`: (inherits false) → `"0"` explicit

**Research sources**:
- Adam L's "avoid" list: "Wipe on external loops make things bad"
- shyblower issue #7326, 2025-02-27: tested `wipe_before_external_loop: true` with scarf seams. "Does not help — same result."

**Why**: locks in the value. Same reasoning as #53.

**Impact**: none.

**Risk**: none.

## Not changing (deliberate)

### `outer_wall_line_width` — stays at 105%

Adam L's guide specifically recommends 150% (0.6 mm on a 0.4 mm nozzle) for "best scarf seam quality" — his reasoning: "wider walls decrease extrusion rate error".

We chose 105% in the earlier full cleanup pass (based on the general Bambu wiki recommendation of 105-120%) as a compromise between scarf quality and dimensional accuracy. The 150% value sacrifices ~0.1 mm of hole dimensional accuracy.

**Decision**: stay at 105%. If this seam tune doesn't fix the visible scarf seam, the next thing to try is raising outer_wall_line_width to 150%. **This would be a separate tuning pass** — isolate one variable at a time.

### `seam_slope_steps` — stays at 20

Adam L says 10 (default) is sufficient; he uses 20 himself with no quality difference. Our 20 is harmless. Not worth changing.

### `scarf_joint_flow_ratio` — stays at default 1.0 (dev mode)

Moved to developer mode in v2.0.0 after Adam L found "tested down to 90% without improvement". Not accessible in normal UI.

**Less likely but possible**: the visible seam was caused by something not covered here (e.g. mechanical Z wobble, filament moisture, drifted PA), and this tuning doesn't fix it. If so, fall back to:

1. Run a PA calibration print for Sunlu PLA Matte on the post-cleanup profile (input shaper was retuned, PA may have drifted)
